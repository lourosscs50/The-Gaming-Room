This repository contains my completed software design document for **The Gaming Room**, created as part of CS 230. The document demonstrates my ability to analyze client requirements, compare operating platforms, make software architecture recommendations, and communicate a technical design clearly to both a client and a development team.

## Client Summary

The client for this project was **The Gaming Room**. The Gaming Room wanted to expand their existing game, **Draw It or Lose It**, from an Android-only application into a web-based, cross-platform application. The software needed to support multiple teams and players, maintain unique game and team names, and provide a reliable experience across different operating platforms.

The main goal was to design a scalable and secure application that could support future growth while remaining accessible to users on multiple devices. This required evaluating operating platforms, memory and storage management, security needs, networking requirements, and deployment options.

## Reflection

### What did I do particularly well in developing this documentation?

One part I did particularly well was organizing the software design document in a way that connected the client’s needs to technical recommendations. I focused on making the design practical and understandable instead of only listing technical details. I also explained the reasoning behind platform recommendations, security considerations, and architecture decisions so that the document could be useful to both the client and a development team.

### What about the process of working through a design document did I find helpful when developing the code?

Working through the design document helped me understand the importance of planning before development. Instead of jumping directly into code, the document forced me to think through the system structure, user requirements, constraints, and possible risks. This made it easier to understand how different parts of the application should work together. It also helped me see how design choices, such as using object-oriented principles and clear class responsibilities, can make the code easier to maintain and expand.

### If I could choose one part of my work to revise, what would I pick, and how would I improve it?

If I could revise one part of my work, I would improve the evaluation of platform trade-offs even further. I would add more detail comparing cloud deployment, Linux server environments, Windows environments, and mobile platforms from a cost, security, and scalability perspective. This would make the recommendation stronger by showing not only which option is best, but also why the other options may be less effective for this specific client.

### How did I interpret the user’s needs and implement them into my software design?

I interpreted the user’s needs by focusing on what The Gaming Room wanted the application to accomplish: support multiple users, allow teams and players, prevent duplicate names, and make the game available beyond a single Android platform. These needs influenced the design by requiring a scalable structure, clear data management, and platform flexibility. Considering the user’s needs is important because software should solve the client’s actual problem, not just meet technical requirements. A design that ignores the user’s goals can result in software that works technically but fails to provide value.

### How did I approach designing software, and what strategies would I use in the future?

My approach to designing software was to first understand the client’s goals, then identify the system requirements, constraints, and best platform options. I used the design document as a planning tool to connect business needs with technical decisions. In the future, I would continue using this type of structured design process. I would also include more diagrams, security planning, scalability analysis, and testing considerations earlier in the project. This would help create a stronger foundation before development begins and reduce the chance of major design changes later.
